## Design Change 2: Lazy Clipboard Stashing

**Before (Design Doc):** Clipboard is stashed eagerly BEFORE the AXUIElement write attempt. Every swap pays the full stash cost (copying all clipboard data via `dataForType:` for every declared type).

**After (this doc):** Clipboard stash is LAZY. Only performed when the AX direct write fails and the Cmd+V fallback path is triggered.

### Updated Swap Pipeline

```
F9 keydown
    │
    ├── isSwapping guard (if true → beep + abort)
    ├── IsSecureEventInputEnabled() (if true → beep + abort)
    │
    ▼
Read text via kAXSelectedTextAttribute
    │
    ├── Empty? → Cmd+Shift+Left fallback → retry read
    ├── Still empty? → beep + abort
    ├── Read-only field? → beep + abort
    ├── > 2000 chars? → beep + abort
    │
    ▼
Translate via TranslationContext
    │
    ▼
Attempt AX direct write (kAXValueAttribute)
    │
    ├── SUCCESS → switch layout → green flash → done
    │
    └── FAILED (sandbox block) ──────────────────────┐
                                                      │
        ┌─────────────────────────────────────────────┘
        │
        ▼
    LAZY CLIPBOARD STASH (only happens here)
    Eager dataForType: copy for all declared types
        │
        ▼
    Write translated text to NSPasteboard
    Poll changeCount (DispatchQueue recursive, 1ms, 50ms timeout)
        │
        ├── Timeout → beep + restore clipboard + abort
        │
        ▼
    Synthetic Cmd+V
    Poll kAXValueAttribute (DispatchQueue recursive, 10ms, 500ms timeout)
        │
        ├── Value changed (paste landed) → restore clipboard → switch layout → green flash → done
        │
        └── Timeout (paste didn't land) → BEEP + restore clipboard + abort
```

**Rationale:** On the happy path (AX direct write succeeds, ~65ms), clipboard is never touched. No memory spike, no risk of clipboard corruption. The stash cost (potentially hundreds of MB for rich clipboard content) is only paid when actually needed.

## Design Change 3: SLA Timeout Replaces Watchdog Timer

**Before (Design Doc):** 2-second `DispatchQueue.main.asyncAfter` watchdog resets `isSwapping` to false as safety net.

**After (this doc):** 500ms SLA timeout. If the swap pipeline exceeds 500ms:
1. Cancel any pending DispatchQueue polls
2. Attempt clipboard restore (best-effort)
3. Emit NSSound.beep()
4. `isSwapping` remains `true` until `defer` fires normally

If the pipeline is truly wedged and `defer` never fires, the user sees "F9 does nothing" and quits from the menu bar. This is safer than resetting `isSwapping` while a stuck swap is still running (which would allow concurrent clipboard corruption).

**Implementation:** Use a `DispatchWorkItem` for the timeout:

```swift
let slaTimeout = DispatchWorkItem { [weak self] in
    self?.cancelPendingPolls()
    self?.restoreClipboard()
    NSSound.beep()
}
DispatchQueue.main.asyncAfter(deadline: .now() + .milliseconds(500), execute: slaTimeout)

// At successful completion:
slaTimeout.cancel()
```

## Design Change 4: Rule 2 Capitalizes First Non-Whitespace Character

**Before (Design Doc):** "If `fallbackMacroUsed == true`, uppercase `output[0]`."

**After (this doc):** "If `fallbackMacroUsed == true`, find the first character matching `[a-z]` in the output and capitalize it."

**Rationale:** `Cmd+Shift+Left` may capture leading whitespace in some apps. `output[0]` could be a space, and uppercasing a space is a no-op. The first actual letter stays lowercase.

**Implementation:**

```swift
if fallbackMacroUsed, let idx = output.firstIndex(where: { $0.isLowercase }) {
    output.replaceSubrange(idx...idx, with: [output[idx].uppercased().first!])
}
```

## Design Change 5: Beep on Failed Paste

**Before (Design Doc):** When kAXValueAttribute polling times out after Cmd+V (500ms), restore clipboard. No explicit failure signal.

**After (this doc):** When polling times out → `NSSound.beep()` + restore clipboard. The beep tells the user the swap failed. Without it, the user sees no change and no feedback — a silent failure.

## Design Change 6: DispatchQueue Recursive Polling

**Before (Design Doc):** "Tight loop" for changeCount polling. "Poll every 10ms" for kAXValueAttribute.

**After (this doc):** Both polling loops use recursive `DispatchQueue.main.asyncAfter` scheduling:

```swift
func pollChangeCount(target: Int, deadline: Date, completion: @escaping (Bool) -> Void) {
    if NSPasteboard.general.changeCount != target {
        completion(true)
        return
    }
    if Date() >= deadline {
        completion(false)
        return
    }
    DispatchQueue.main.asyncAfter(deadline: .now() + .milliseconds(1)) { [weak self] in
        self?.pollChangeCount(target: target, deadline: deadline, completion: completion)
    }
}
```

**Rationale:** No CPU spin. No `Thread.sleep`. The main run loop stays responsive between checks. Cooperative scheduling.

- changeCount polling: 1ms interval, 50ms timeout
- kAXValueAttribute polling: 10ms interval, 500ms timeout

## Design Change 7: IOHIDRequestAccess Verification

**Before (Design Doc):** Onboarding flow uses `IOHIDRequestAccess(kIOHIDRequestTypeListenEvent)` for Input Monitoring permission.

**After (this doc):** Mark as VERIFY ON MAC. `IOHIDRequestAccess` may not be a reliable public API. Some sources indicate Input Monitoring permission is implicitly triggered when creating a `CGEventTap`.

**Verification plan:** During Phase 2, test on Mac:
1. Call `IOHIDRequestAccess(kIOHIDRequestTypeListenEvent)` — does it compile? Does it trigger the system dialog?
2. If yes: proceed as designed
3. If no: redesign onboarding to explain that the system permission dialog appears automatically when KeySwap first tries to monitor keyboard input. The "Grant Input Monitoring" button becomes "Open System Settings → Input Monitoring" with instructions to add KeySwap manually.

### SEC-1: CGEventTap Keystroke Filtering (CRITICAL)

**Risk:** The CGEventTap callback receives EVERY keyboard event system-wide. If the filter has a bug (e.g., falls through to processing logic for non-F9 keys), KeySwap could inadvertently read, log, or act on arbitrary keystrokes.

**Mitigation (mandatory):**

```swift
// GlobalHotkeyListener.swift — CGEventTap callback
//
// SECURITY: This callback receives ALL keyboard events.
// Return the event IMMEDIATELY for anything that is not
// exactly keyCode 100 (F9), with or without Shift.
// Do NOT log, inspect, store, or process non-F9 events.

func eventTapCallback(proxy: CGEventTapProxy,
                      type: CGEventType,
                      event: CGEvent,
                      refcon: UnsafeMutableRawPointer?) -> Unmanaged<CGEvent>? {

    // SECURITY GATE: only process F9 (keyCode 100)
    guard type == .keyDown else {
        return Unmanaged.passRetained(event)  // pass through
    }

    let keyCode = event.getIntegerValueField(.keyboardEventKeycode)
    guard keyCode == 100 else {
        return Unmanaged.passRetained(event)  // pass through immediately
    }

    // Only F9 / Shift+F9 reaches here
    // ... swap logic ...
}
```

**Rules:**
- The `guard keyCode == 100` check MUST be the first logic after the event type check. No logging, no variable assignment, nothing before this gate.
- Non-F9 events MUST be returned via `Unmanaged.passRetained(event)` with zero processing.
- No `print()`, `os_log()`, or any form of output for non-F9 events. Ever.

**Test:** Unit test that mocks the callback with keyCode != 100 and asserts the event is returned immediately with no side effects.

### SEC-2: AXUIElement Read Scope (HIGH)

**Risk:** `AXUIElementCopyAttributeValue` with `kAXSelectedTextAttribute` reads whatever text is selected in the focused app. If `IsSecureEventInputEnabled()` has gaps (some apps or custom controls don't set the secure input flag for sensitive fields like API key inputs, 2FA codes, or credit card forms), KeySwap could read sensitive content.

**Mitigation:**
1. `IsSecureEventInputEnabled()` check runs BEFORE any AXUIElement read. This is already in the design.
2. The text read from AXUIElement is held only in a local variable within the swap function scope. It is NEVER:
   - Written to disk
   - Logged (not even in debug builds)
   - Stored in any instance variable that outlives the function
   - Sent over any network connection
3. The translated output replaces the input in the same scope. After `defer` fires, both are deallocated.

**Rule:** Add a code comment at the AXUIElement read site:

```swift
// SECURITY: selectedText contains user content from the focused app.
// It may contain sensitive data if IsSecureEventInputEnabled() has gaps.
// This variable MUST NOT be logged, persisted, or stored beyond this scope.
let selectedText = // ... AXUIElement read
```

**Test:** Code review checklist item. grep the codebase for any `os_log`, `print`, `NSLog`, or `Logger` call that references the selected text variable.

### SEC-3: Clipboard Stash Data in Memory (MEDIUM)

**Risk:** During the Cmd+V fallback path (~500ms), the clipboard contents are held in a local dictionary. If the user had a password, auth token, or sensitive data in their clipboard, it lives in process memory during this window. If the process crashes, this data could appear in a crash report or core dump.

**Mitigations:**
1. **Lazy stash** (Design Change 2) already limits exposure. Clipboard data is only copied on the fallback path, not every swap.
2. **Scope the stash variable tightly.** The `ClipboardSnapshot` dictionary should be a local variable inside the fallback function, not an instance property on ClipboardManager. When the function returns (success or failure), the data is deallocated.
3. **Zero the Data buffers on restore.** After restoring the clipboard, overwrite the stashed `Data` objects with zeros before releasing them:

```swift
// SECURITY: Zero sensitive clipboard data after restore
for (_, var data) in stashedItems {
    data.resetBytes(in: 0..<data.count)
}
// stashedItems goes out of scope here
```

4. **Crash report exclusion.** macOS crash reports include register state and some stack memory, but not the full heap. The stashed data is heap-allocated, so it won't appear in standard crash reports. No additional mitigation needed for this vector.

**Residual risk:** If the process is inspected by a debugger or memory forensics tool during the 500ms window, the clipboard data is readable. This is acceptable: any app with Accessibility permissions has this property. The user already granted trust.

### SEC-4: Hardened Runtime Entitlements (MEDIUM)

**Risk:** The Design Doc says "verify at build time whether `com.apple.security.cs.disable-library-validation` is needed for the CGEventTap." This entitlement disables library validation, allowing unsigned dylibs to be loaded into the process. If granted, a malicious dylib could be injected to intercept keystrokes or clipboard data.

**Mitigation:**
1. **Do NOT use `cs.disable-library-validation` unless CGEventTap absolutely requires it.** Test on Mac during Phase 2 without this entitlement first.
2. The only entitlement that should be needed is `com.apple.security.automation.apple-events` (for Accessibility API + synthetic events).
3. If `disable-library-validation` IS required (unlikely for session-level taps), document exactly why and add a code comment in the entitlements file explaining the security tradeoff.

**Verification step (Phase 2):**
```bash
# Build with hardened runtime, WITHOUT disable-library-validation
# Test CGEventTap creation. If it works: done. If it fails: investigate.
codesign --verify --deep --strict KeySwap.app
```

### SEC-5: Translation Input Validation (LOW)

**Risk:** Text read from AXUIElement is passed directly to TranslationContext and then written back (AX write or clipboard). If the text contains control characters, null bytes, or Unicode edge cases, the write-back could behave unexpectedly in certain target apps.

**Mitigation:**
1. TranslationContext already handles the character mapping: unmapped characters pass through unchanged. This is correct behavior, the translation layer should not strip or modify characters it doesn't understand.
2. **RTL stripping** (already in design) removes direction markers that could cause layout glitches.
3. **No additional sanitization needed.** The text came from the app and goes back to the same app. We are a transparent pipeline, not a sanitizer. Stripping control characters could corrupt legitimate content (e.g., tab characters in a code editor).

**Residual risk:** Acceptable. The text was already in the app. We're not introducing new content.

### SEC-6: .dmg Distribution Security (LOW)

**Risk:** The .dmg is distributed outside the Mac App Store via direct link. If the hosting is compromised, an attacker could replace the .dmg with a malicious version that has Accessibility + Input Monitoring permissions (the user would grant them thinking it's KeySwap).

**Mitigations (already in design):**
1. **Code signing** with Apple Developer ID. macOS Gatekeeper verifies the signature before allowing installation. A replaced .dmg would have a different signature and Gatekeeper would block it.
2. **Notarization** via `xcrun notarytool`. Apple scans the binary for malware and issues a ticket. macOS checks the ticket on first launch.
3. **Stapling** attaches the notarization ticket to the .dmg so verification works offline.

**Additional recommendation:** When distributing, publish the SHA-256 hash of the .dmg alongside the download link so users can verify integrity independently.

### SEC-7: No Telemetry / No Network (ENFORCED)

**Risk:** Apps with keystroke monitoring capability are high-value targets for scope creep. "Just add a crash reporter" or "just add analytics" could create an exfiltration channel.

**Rule (absolute, for all versions):**
- Zero network calls. No crash reporters (Sentry, Crashlytics, etc.). No analytics. No update checks.
- `TranslationContext` must never import Foundation networking types (`URLSession`, `URLRequest`).
- The daemon must never open a socket or resolve a DNS name.

**Verification:** During code review, grep for network-related symbols:

```bash
# Run during Phase 3 code review
grep -rn "URLSession\|URLRequest\|URL(string\|HTTPURLResponse\|socket\|NWConnection\|Network.framework" Sources/
# Expected result: zero matches
```

**Test:** Add a build-time script or unit test that asserts no networking symbols are linked.
