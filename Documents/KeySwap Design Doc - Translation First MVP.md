## Problem Statement

Bilingual Hebrew/English Mac users frequently type entire sentences in the wrong keyboard layout. The result is gibberish. The fix today is: delete everything, switch layout, retype from memory. This breaks flow, wastes time, and happens dozens of times per day for fast typists.

KeySwap solves this with a single F9 press: detect the selected (or current-line) text, swap every character to the correct layout via a bidirectional mapping table, and inject the corrected text back — all while preserving whatever was in the clipboard.

## What Makes This Cool

1. **Clipboard preservation via eager stashing.** No competitor does this. Most layout-correction tools destroy your clipboard on every swap. Before each swap, KeySwap eagerly reads all type+data pairs from the current `NSPasteboardItem` array, stores them locally, performs the swap, then restores the original clipboard contents. This is a transient memory spike (bounded by whatever was in the clipboard) lasting < 500ms, after which stashed data is released.

2. **The V2.0 vision: auto-detection (out of scope for MVP).** The builder's real ambition is an app that watches you type and silently fixes wrong-layout input without any hotkey. That's magic. The MVP nails the manual trigger; V2.0 removes the trigger entirely.

3. **Hebrew/macOS is underserved.** Most competitors target Russian/English. The Hebrew niche has no well-crafted native solution.

## Constraints

- Native Swift 5+ / AppKit only. No external dependencies. No network calls.
- Background daemon (LSUIElement = YES). Resting memory < 30MB.
- End-to-end execution latency < 500ms (F9 keydown to text injection).
- macOS Accessibility + Input Monitoring permissions required.
- Must run outside App Store sandbox (direct .dmg distribution) because Accessibility API access is incompatible with App Store sandboxing.
- 2,000 character payload limit per swap.
- No persistent state — all execution state is transient in-memory.

## Premises

1. **Distribution:** Ship as direct .dmg download, not Mac App Store. Sandboxing breaks Accessibility APIs.
2. **Fallback selection:** Cmd+Shift+Left (select whole line) is acceptable for MVP. "Greedy line detection" (finding where wrong layout begins) is deferred to post-MVP.
3. **Feedback:** Error auditory feedback (NSSound.beep) + visual success flash (green menu bar icon for 0.5s) for MVP.
4. **Auto-detection:** V2.0 feature. MVP uses F9 hotkey trigger only.
5. **RTL handling:** Must strip Unicode RTL/LTR direction markers (\u200F, \u200E, \u202B, \u202C) during translation to prevent visual glitches when swapping Hebrew-layout text to English. Handle in the translation layer, not the injection layer.
6. **Per-app context awareness:** V2.0 feature. MVP operates globally.

## Recommended Approach

**Approach A: Translation-First.** Build and test the translation engine in complete isolation as a Swift Package, then wire in the macOS daemon shell.

### Phase 1: TranslationContext Swift Package
- Create a pure Swift Package (no AppKit dependency)
- Implement the full English/Hebrew bidirectional character mapping table (see Character Mapping Reference below)
- Implement the 5 capitalization/mapping rules:
  - Rule 0 (Gate): Auto-capitalization rules execute ONLY when the target output language is English. When target is Hebrew, skip rules 1-3.
  - Rule 1 (Explicit Pass-Through): Any `[A-Z]` character in the input maps directly to `[A-Z]` in the output.
  - Rule 2 (Line Replacement): If `fallbackMacroUsed == true` (line was auto-selected), uppercase `output[0]`.
  - Rule 3 (Punctuation Check): Regex lookbehind `(?<=[.!?]\s)[a-z]` — capitalize after sentence-ending punctuation. Applied to the **translated output string** (not the input), after Rules 1-2 have been applied. Applies only to characters inside the selection.
  - Rule 4 (Default): All other characters default to lowercase English or standard Hebrew.
  - **Application order:** Rule 0 (gate) -> Rule 1 (applied character-by-character during translation) -> Rule 4 (default, during translation) -> Rule 2 (post-translation, applied to output[0]) -> Rule 3 (post-translation, regex on full output string).
- Implement RTL unicode mark stripping (CharacterSet filter for \u200F, \u200E, \u202B, \u202C). By the time text leaves TranslationContext, it is guaranteed RTL-direction-marker-clean. Downstream injection layers (AXUIElement write or Cmd+V paste) never need to handle RTL stripping.
- Write comprehensive XCTest suite covering:
  - Hebrew-to-English with all 5 capitalization rules
  - English-to-Hebrew
  - Mixed punctuation edge cases
  - RTL marker stripping
  - Empty string, single character, max payload (2000 chars)
  - Characters not in the mapping table (pass-through)
  - XCTest performance measure block asserting translation of a 2000-char string completes in < 10ms

### Translation Direction Detection (Keyboard-Layout-Based)

Direction detection is NOT part of TranslationContext. It lives in the daemon shell (Phase 2) because it requires macOS APIs. TranslationContext takes direction as a parameter (`targetLanguage: .english` or `.hebrew`).

The daemon detects swap direction using the current keyboard layout:
1. Call `TISCopyCurrentKeyboardInputSource()` to get the active input source.
2. Read `kTISPropertyInputSourceID` from the returned source.
3. If the input source ID contains "Hebrew" → swap Hebrew→English. Target: English.
4. If the input source ID does NOT contain "Hebrew" (English or other) → swap English→Hebrew. Target: Hebrew.
5. After successful swap, switch the keyboard layout to the target language via `TISSelectInputSource()`.
6. If no text to swap (empty selection, no mappable characters) → do not switch layout.

**Why keyboard-layout-based instead of text analysis:** The previous approach (Unicode-range heuristic) would misidentify punctuation-only English text because characters like comma and period exist in the Hebrew mapping table. Keyboard layout is unambiguous: the user's current layout tells you exactly which direction they need to go. It also enables a natural UX: after swapping, the keyboard is already in the correct layout for continued typing.

### Phase 2: macOS Daemon Shell
- `Info.plist` — LSUIElement = YES
- `KeySwapApp.swift` — silent boot, NSStatusItem menu bar. Call SMAppService.mainApp.register() immediately on first launch (before onboarding). If registration fails, log the error and continue (login item is a convenience, not a requirement for MVP). Menu bar menu items in ACTIVE state: "KeySwap — Active" (disabled label), separator, "About KeySwap", "Quit KeySwap". In DEGRADED state: "KeySwap — Degraded (retrying...)" (disabled label, warning icon), separator, "About KeySwap", "Quit KeySwap".
- `AppState.swift` — permission tracking state machine. Both Accessibility and Input Monitoring grants are independent (either can be granted first). States: PERMISSIONS_REQUIRED (neither granted), PARTIAL (one of two granted), ACTIVE (both granted), DEGRADED (CGEventTap failed or disabled by macOS). Transitions: ACTIVE → DEGRADED on `CGEventTapIsEnabled()` returning false or tap callback not firing. Recovery: 30-second retry loop attempts `CGEventTapEnable(tap, true)`. If re-enable succeeds: DEGRADED → ACTIVE. Menu bar in DEGRADED: warning icon + "KeySwap — Degraded (retrying...)". Track transition timestamps for conversion metric.
- `PermissionsRouter.swift` — onboarding window with the following flow:
  1. On first launch, show a single-window explanation of why permissions are needed
  2. Button: "Grant Accessibility" -> calls AXIsProcessTrusted(). If not trusted, opens System Settings -> Privacy -> Accessibility. Poll AXIsProcessTrusted() every 1s until granted or window closed.
  3. Button: "Grant Input Monitoring" -> calls IOHIDRequestAccess(kIOHIDRequestTypeListenEvent). System shows its own permission dialog.
  4. If user closes window without granting both: AppState remains PERMISSIONS_REQUIRED. Menu bar shows "Status: Permissions Required" with option to reopen onboarding.
  5. If both granted: AppState transitions to ACTIVE. Onboarding window closes.
- `GlobalHotkeyListener.swift` — CGEventTap for F9 (keyCode 100) and Shift+F9 (keyCode 100 + Shift modifier) using `kCGSessionEventTap` + `kCGHeadInsertEventTap` (session-level tap, does not require root, works with Input Monitoring permission). Both F9 and Shift+F9 trigger the identical swap pipeline. Shift+F9 exists as a mental "undo swap" for non-technical users, though both keys do the same thing (direction is determined by keyboard layout, so pressing either key again swaps back). On trigger: check `isSwapping` re-entrancy guard first (if true, beep+abort). Then check `IsSecureEventInputEnabled()`. If true: emit NSSound.beep() and abort.
- `AccessibilityInteractor.swift` — kAXSelectedTextAttribute read, kAXValueAttribute write. If AXUIElementSetAttributeValue returns kAXErrorNotImplemented or kAXErrorCannotComplete (target app rejects direct text write, common in Electron apps and some sandboxed Mac App Store apps), fall back to ClipboardManager + Cmd+V injection path.
- `ExecutionProfile.swift` — writable field check, 2000 char cap
- `SyntheticEventGenerator.swift` — Cmd+Shift+Left (fallback selection), Cmd+V (paste fallback)
- `ClipboardManager.swift` — clipboard preservation and injection fallback:
  1. **Stash:** Capture `NSPasteboard.general.changeCount`. For each `NSPasteboardItem`, eagerly read `dataForType:` for every declared type and store the type+data pairs in a local dictionary. This is critical: holding a reference to the NSPasteboardItem object alone does NOT survive `clearContents()` because the pasteboard server evicts backing data when cleared. Eager copy at stash time is the only reliable approach. Memory impact: bounded by the 2000-char selection limit (the stashed clipboard data may be larger, but this is a transient spike lasting < 500ms).
  2. **Write payload:** Clear the general pasteboard and write the translated string.
  3. **Verify:** Poll `NSPasteboard.general.changeCount` in a tight loop (max 50ms timeout) until the count increments, confirming the clipboard server registered the new payload.
  4. **Inject:** Fire synthetic Cmd+V via CGEvent.
  5. **Restore:** After Cmd+V, poll the target AXUIElement's `kAXValueAttribute` every 10ms. When the value changes (paste has landed), immediately restore clipboard. Timeout at 500ms if no change detected — restore clipboard anyway + log warning. This replaces a fixed 50-100ms sleep with deterministic polling.
  6. **Failure:** If any step times out, emit NSSound.beep(). Do NOT leave the pasteboard in a corrupted state — always attempt restoration even on failure.

- `LayoutSwitcher.swift` — Keyboard layout detection and switching:
  1. **Detect:** `TISCopyCurrentKeyboardInputSource()` → read `kTISPropertyInputSourceID` → determine if Hebrew or English.
  2. **Switch:** After successful swap, call `TISSelectInputSource()` with the target layout. Requires building a reference to the target input source via `TISCreateInputSourceList()` filtered by source ID.
  3. **Error:** If `TISSelectInputSource()` fails, log warning but do not abort (swap already succeeded, user can switch manually).

- `AboutWindow.swift` — Native NSWindow or SwiftUI sheet:
  - App name + version (from `Bundle.main.infoDictionary`)
  - One-sentence description: "Corrects bilingual Hebrew/English typing errors"
  - Hotkey reminder: "Press F9 to swap. Press Shift+F9 to swap back."
  - Link to report issues (opens URL in default browser)
  - Accessible from menu bar: "About KeySwap" menu item

- **Visual Success Flash:** On successful swap completion, briefly change the `NSStatusItem.button?.image` to a green variant for 0.5 seconds, then revert to default via `DispatchQueue.main.asyncAfter(deadline: .now() + 0.5)`. No animation framework needed.

- **Re-entrancy Guard:** Boolean `isSwapping` flag checked at F9/Shift+F9 entry point. If `isSwapping == true` → beep + abort. Set `isSwapping = true` at entry, `defer { isSwapping = false }` for guaranteed cleanup. Additionally, a 2-second watchdog timer (`DispatchQueue.main.asyncAfter(deadline: .now() + 2.0)`) force-resets `isSwapping` to false as a safety net if the swap crashes mid-execution and `defer` doesn't fire.

## Latency Budget (worst case: Cmd+V fallback path)

| Step | Estimated Time |
|------|---------------|
| CGEventTap dispatch + IsSecureEventInputEnabled() | ~5ms |
| AXUIElement read (kAXSelectedTextAttribute) | ~20ms |
| AX write attempt + failure detection | ~30ms |
| TranslationContext execution (2000 chars) | < 10ms |
| ClipboardManager stash (eager copy) | ~20ms |
| Pasteboard write + changeCount polling | ~50ms |
| Cmd+V synthetic event + paste processing | ~100ms |
| Clipboard restore (poll-based, up to 500ms timeout) | ~50ms typical |
| Keyboard layout switch (`TISSelectInputSource`) | ~5ms |
| Visual success flash trigger | ~1ms |
| **Total worst case** | **~291ms** |
| **Headroom** | **~209ms** |

The 500ms SLA is comfortably met even on the fallback path. The AXUIElement direct-write path (no clipboard involvement) is significantly faster (~65ms).

## Error States & Fallbacks

| Trigger | Condition | Action |
|---------|-----------|--------|
| F9 pressed | IsSecureEventInputEnabled() == true | NSSound.beep(), abort |
| F9 pressed | AppState != ACTIVE (permissions not granted) | NSSound.beep(), abort |
| Text extraction | kAXSelectedTextAttribute returns nil/empty AND fallback Cmd+Shift+Left also yields empty | NSSound.beep(), abort |
| Text extraction | Selected text > 2000 characters | NSSound.beep(), abort |
| Text extraction | Target AXUIElement is read-only (`AXUIElementIsAttributeSettable(element, kAXValueAttribute, &settable)` returns false) | NSSound.beep(), abort |
| Text injection | AXUIElementSetAttributeValue fails (sandbox block) | Fall back to ClipboardManager + Cmd+V path |
| Clipboard | changeCount polling times out (50ms) during payload write | NSSound.beep(), abort, attempt clipboard restoration |
| Clipboard | Restoration after injection fails | Log error silently (user's clipboard is lost for this operation — acceptable edge case) |
| Translation | Input contains only unmapped characters | Return input unchanged, no beep (silent no-op) |
| F9/Shift+F9 | `isSwapping == true` (re-entrancy) | NSSound.beep(), abort |
| Text extraction | AXUIElement not obtainable (no focused element or no AX support) | NSSound.beep(), abort |
| Clipboard stash | `dataForType:` returns nil for a declared type | Skip that type, continue stash with remaining types (partial stash) |
| Synthetic event | `CGEvent(keyboardEventSource:...)` returns nil | NSSound.beep(), abort |
| CGEventTap | Tap disabled by macOS or fails to create | Transition to DEGRADED state, start 30s retry loop |

### Resolved Decisions
- **Auto-detection architecture (V2.0):** Decision: keep V1.0 completely clean. Do NOT add keystroke pattern capture to the CGEventTap. Add the hook in V2.0 planning. Rationale: premature hooks add complexity with zero MVP value.
- **Swap history log:** Deferred to V2.0.
- **Direction detection method (CEO review):** Changed from Unicode-range heuristic to keyboard-layout-based (`TISCopyCurrentKeyboardInputSource()`). The heuristic had a punctuation ambiguity bug (commas/periods exist in the Hebrew mapping table, so English text with only punctuation would be misidentified). Layout-based detection is simpler and unambiguous.
- **Clipboard restore timing (CEO review):** Changed from fixed 50-100ms sleep to AXUIElement `kAXValueAttribute` polling with 500ms timeout. Deterministic, no wasted time, no risk of restoring too early.
- **Re-entrancy guard (CEO review):** Added `isSwapping` boolean with `defer` cleanup and 2-second watchdog timer. Prevents double-tap F9 double-swap and guards against stuck flag after crash.
- **Shift+F9 swap-back (CEO review):** Added as explicit swap-back trigger. Functionally identical to F9 (bidirectional map + layout detection makes direction automatic). Provides mental safety net for non-technical users.
- **Visual success feedback (CEO review):** Promoted from deferred to MVP scope. Green menu bar icon flash for 0.5s on successful swap. Tiny effort, meaningful UX for non-technical audience.
- **About window (CEO review):** Added to MVP scope. Version, F9/Shift+F9 hotkey reminder, one-sentence description, issue report link.
- **DEGRADED state (CEO review):** Added to AppState machine. CGEventTap failure → DEGRADED → 30s retry → ACTIVE. Menu bar shows warning.
